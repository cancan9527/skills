# Workflow-Heavy PRD Guidelines

## Purpose

This reference strengthens PRD generation for workflow-heavy requirements such as:

- ERP flows
- MES flows
- approval processes
- order or document flows
- operations-heavy back-office systems
- multi-role business processes with status transitions

Use it when the feature is not just a page or form, but a process with roles, states, actions, exceptions, and writebacks.

The goal is to prevent the PRD from becoming “page-complete but rule-incomplete”.

## When to Use This Reference

Use this reference when the requirement includes one or more of these traits:

- multiple roles participate in sequence
- a parent object has child stages or sub-flows
- state transitions matter
- actions depend on role or state
- completion of one step activates another step
- data must be written back to upstream or downstream records
- to-dos, notifications, or approvals are triggered by events

If these traits are present, strengthen the PRD instead of relying on a lightweight page-description style.

## Core Generation Principle

For workflow-heavy requirements, do not stop at:

- page list
- button names
- simple flow summary

The PRD must also explain:

- who acts
- when they can act
- what state they act from
- what changes after the action
- what can go wrong
- what is written back

## Required Reinforcement Areas

### 1. Roles and Permissions

The PRD should not merely list roles. It should explain role responsibility boundaries.

Minimum coverage:

- initiating role
- handling role for each stage
- approval or confirmation role
- read-only roles
- admin or exception-handling role when relevant

Recommended table:

| 角色 | 可查看 | 可新增 | 可编辑 | 可提交 | 可审批/确认 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |

Typical omissions:

- only naming the role, not the authority
- no distinction between owner and approver
- no rule for view-only participants

### 2. Main Flow Closure

The PRD should define a complete normal path, not just a list of stages.

Minimum coverage:

- start condition
- step-by-step actions
- system response after each step
- handoff between roles
- completion condition

Recommended structure:

1. trigger
2. actor
3. action
4. system response
5. next state or next stage

Typical omissions:

- middle steps are implied but not written
- handoff between two roles is unclear
- process appears complete visually but lacks closure in rules

### 3. Exception and Reverse Flows

Workflow-heavy PRDs must explicitly cover non-happy paths.

Minimum coverage:

- reject
- return
- withdraw
- cancel
- rollback
- partial success
- repeated submission
- conflict with current state

Ask and write:

- can this step be reversed?
- who can reverse it?
- until what point can it be reversed?
- what data or state is restored after reversal?

Typical omissions:

- only success flow is written
- error prompt exists but no handling rule
- reversal exists in prototype but not in PRD

### 4. State Definitions

Every important business object should have clear state definitions.

Minimum coverage:

- state name
- business meaning
- entry condition
- exit condition
- whether it is terminal

Recommended table:

| 状态 | 业务含义 | 进入条件 | 退出条件 | 是否终态 |
| --- | --- | --- | --- | --- |

Typical omissions:

- status names exist but business meaning does not
- same object has parent status and child status but relation is unclear
- state list exists without entry/exit rules

### 5. Button and Action Rules

Buttons should be treated as business actions, not just UI controls.

Minimum coverage:

- action name
- visible states
- enabled states
- executable roles
- precondition
- post-action state
- whether confirmation is required

Recommended table:

| 动作/按钮 | 可见条件 | 可执行角色 | 前置条件 | 执行结果 | 后置状态 |
| --- | --- | --- | --- | --- | --- |

Typical omissions:

- “show button” is written but “what it changes” is not
- hidden vs disabled is not distinguished
- action allowed in prototype but blocked in business logic

### 6. Business Object Relationships

Workflow-heavy systems usually contain parent-child or upstream-downstream relationships.

The PRD should explain:

- main object
- child object
- generated object
- linked object
- relation type

Typical examples:

- one request generates many sub-records
- one failed inspection creates a new rework record
- one approval updates both parent and child objects

Minimum coverage:

- which object is the source of truth
- which object is generated by which event
- whether repeated generation must be prevented

### 7. Data Writebacks

A workflow action is incomplete unless its data consequences are clear.

Minimum coverage:

- which fields are updated
- which object receives the update
- when the update occurs
- whether the update is synchronous or deferred
- whether upstream or downstream records also change

Recommended table:

| 触发动作 | 更新对象 | 更新字段 | 写回目标 | 时机 | 备注 |
| --- | --- | --- | --- | --- | --- |

Typical omissions:

- page says “complete” but no field update is defined
- status changes visually but not in source-of-truth data
- cross-system updates are implied but not written

### 8. To-Dos, Notifications, and Approvals

If the feature triggers to-dos, notifications, or approvals, the PRD should define them as business rules.

Minimum coverage:

- trigger condition
- recipient
- content or type
- whether it can repeat
- whether it closes automatically
- whether business status depends on it

Typical omissions:

- “send notification” is written without recipient rule
- duplicate triggers are not prevented
- to-do completion does not say whether it changes business flow

### 9. Time Recording and Traceability

Workflow-heavy features often need auditability and timing analysis.

Minimum coverage:

- which action writes start time
- which action writes end time
- confirmation or approval timestamps
- operation logs or change records

Typical omissions:

- business says timeliness matters, but no timestamp rule is defined
- end state exists with no recorded end time

### 10. Acceptance Criteria

Workflow-heavy acceptance must validate process logic, not only UI behavior.

Minimum coverage:

- normal flow case
- exception flow case
- role restriction case
- state transition case
- writeback case
- notification or to-do case when relevant

Recommended grouping:

- normal scenarios
- exception scenarios
- permission scenarios
- state scenarios
- linkage scenarios

## Parent and Child State Guidance

If the feature has both a parent process and child sub-processes, define both levels.

Clarify:

- whether parent status is derived from child statuses
- whether any child completion activates the next stage
- whether all children must complete before parent can advance
- what happens if one child fails and others succeed

This is a frequent weak point in process-heavy PRDs and should be written explicitly.

## Generation Heuristics

When generating a workflow-heavy PRD:

1. prefer tables over vague prose for rules
2. define state and action semantics explicitly
3. separate main flow from exception flow
4. separate UI action from business consequence
5. surface assumptions in `待确认问题` instead of silently inventing logic

If a critical rule is still unknown, do not fabricate certainty. Draft the structure and flag the gap clearly.

## Common Failure Patterns

Watch for these failure patterns when drafting:

- page-complete, process-incomplete
- statuses listed, transitions missing
- buttons listed, role restrictions missing
- actions defined, writebacks missing
- child flows described, parent summary missing
- notifications mentioned, recipients or triggers missing
- completion claimed, acceptance criteria missing

If you see these patterns in the input, reinforce them in the PRD draft instead of mirroring the weakness.
